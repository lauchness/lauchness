In the last post, [how to useReducer](/blog/how-to-use-reducer), I went over the
basics of implementing a useReducer hook in React, and some basics of managing
state in React in general. In this post, we'll pick up where we left off and
talk about how useReducer can help manage state **accurately**.

Our components that implement useReducer, will usually be managing complex
state, that likely has a number of properties. Since we have so many properties,
we actually have many _possible states_, but that doesn't mean all of them are
valid.

What do I mean by a **valid state**? Well, let's take a look at the state object
we had in the last post.

```javascript
{
  list: [{...}, {...}],
  description: 'javascript',
  location: 'new york',
  fullTime: false,
  loading: false
}
```

In this example, we have a job search tool, which manages the parameters:
location, job description, and full-timieness of the jobs; as well as a loading
state, and the list of jobs related to our search parameters.

Imagine for a moment that we have an action for each slice of state:
`SET_FULL_TIME`, `SET_DESCRIPTION`, `SET_LOADING`, and so on. This would mean
that we could transition between every possible state (if you're interested, it
would be a [cartesian product](https://en.wikipedia.org/wiki/Cartesian_product)
of all the state properties). But not all of these are **valid states**, for
example, if we have `loading: true`, we probably want our list of jobs to be an
empty array.

This is where we want to start thinking about our reducer as a state machine.

## Finite State Machine

A
[Finine State Machine](https://en.wikipedia.org/wiki/Finite-state_machine)(FSM)
is a model that can be used to manage states. The term makes it sound much more
complex than it really is. What the FSM does is provide us (the developers) with
a map that we can use to transition between valid states in our application.

An FSM will have an entry point (or initial state), and a map between possible
states based on inputs we receive from user interaction. We will treat our user
actions as useReducer actions (makes sense right!).

There are schools of thought on how these maps should be drawn, and I encourage
you to follow the rabbit hole and learn all about state machines.

## Flux Pattern

As we just discussed, all of the state in a React app flows down through the
tree structure. This is all well and good in theory, but let's take a look at an
example from our Job Filter scenario above.

Imagine we're happily managing our `fullTime` state within our Job Filter
component, as things should be. But we get a ticket from on high which demands
that we make our toggle input a distinct component. Well, now that new component
will need to receive the `fullTime` state as a prop, AND it will need to be able
to manage that state. To accomplish that we'll need to pass down a callback
function as a prop which can set the state in the parent, where it's been
declared.

```jsx
const JobFilter = () => {
  const initialFullTime = false

  const [fullTime, setFullTime] = useState(initialFullTime)

  return (
    <>
      // some bunch of JSX
      <FullTimeToggle fullTime={fullTime} setFullTime={setFullTime} />
    </>
  )
}
```

So here, we can clearly see the fundamentals of the "Flux Pattern". Flux was
developed at Facebook for managing the "unidirectional" data flow of React apps.

In a Flux Pattern, there are three major parts: the dispatcher, the stores, and
the views. While we're not implementing the flux library in the above example,
the pattern has really worked it's way into the core API of hooks:

- The **Store** is the `fullTime` value
- The **Dispatcher** is the `setFullTime` callback function
- The **View** is the JSX which relies on the `fullTime` value to render, and
  can call the `setFullTime` function from a user interaction

While this example is somewhat simplified, it clearly outlines the pattern. All
of our data flows down from some store, and is modified by dispatching updates.

## Implementing useReducer

We know that our job filter actually has more than a `fullTime` state, it has a
whole object full of state to manage. Rather than adding a bunch of distinct
useState hooks, we can implement the useReducer hook. Before doing that though,
let's compare the useReducer API to the useState API and try to find all those
same pieces of the Flux Pattern.

When we call a useReducer hook within a component, it takes a reducer function
and an initial state as input. It then returns a state value and a dispatch
function.

```jsx
const JobFilter = () => {
  const myReducer = (state, action) => {
    // We'll write this shortly
  }

  const initialState = {
    list: [],
    description: "",
    location: "",
    fullTime: false,
    loading: false,
  }

  const [state, dispatch] = useReducer(myReducer, initialState)

  const setFullTime = dispatch(/* we'll write this shortly */)

  return (
    <>
      // some bunch of JSX
      <FullTimeToggle fullTime={fullTime} setFullTime={setFullTime} />
    </>
  )
}
```

> Note: You can pass an
> [init function](https://reactjs.org/docs/hooks-reference.html#lazy-initialization)
> to create your initial state lazily, but we'll ignore that for now

If we really take a close look at this API and the useState API, the only real
difference is the use of a reducer function. Otherwise, we can break it down the
same way:

- The **Store** is the `state` value
- The **Dispatcher** is the `dispatch` callback function
- The **View** is the JSX which relies on the `state` value to render, and can
  call the `dispatch` function from a user interaction

I like to think of useState as a shortcut for a simple case of useReducer. So,
reading the state is simple enough, we just access the property we want (i.e.
`state.fulltime`) and use it. The question now is how do we update that state?
In a useState hook, we would call the setState function with the new state we
want. useReducer is really quite similar, only this time we'll be calling
dispatch with what we call an **action**.

Actions are objects get consumed by reducers to produce a new state value. The
common pattern for action objects is to have a `type` property so the reducer
knows which slice of state to update, and a `payload` so the reducer can
manipulate that slice of state (e.g. inserting an item into an array, replacing
a value, adding to a counter, etc.).

In our job filter example we could have a `setFullTime` action that looks like
this:

```javascript
const setFullTime = {
  type: "SET_FULL_TIME",
  payload: true,
}

dispatch(setFullTime)
```

Now that we know what an action is, we need to understand how that gets consumed
by a reducer. When the action is dispatched, a reducer function we will write
receives the current state and the action we supply as inputs. The reducer
function needs to do some work with these inputs (which we'll talk about in a
moment), and return the new state.

It's common for the reducer to use a
[switch statement](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/switch)
and define a case for every action type we want to use. Originally, I had
written that a useReducer should return the original state in the default case
of our switch statement. However [@kv385191](https://twitter.com/kv385191)
pointed out this tweet to me:

https://twitter.com/dan_abramov/status/1093969005675773954

This is a prime example of having "strong opinions, loosely held". I am
convinced this makes sense. If our reducer runs across an `action.type` that we
didn't define, then our code has hit a bug. So let's throw an error for the
default state!

If do this with the reducer from our example above, it could look like this.

```javascript
const myReducer = (state, action) => {
  switch (action.type) {
    default:
      throw new Error(
        `Action type: ${action.type} was not accounted for in myReducer`
      )
  }
}
```

Since we're all super cool, and have real state to manage, we will want to
respond to our actions. To do that we'll need to define cases for our action
types.

If we implement a case for the `setFullTime` action it could look like this.

```javascript
const myReducer = (state, action) => {
  switch (action.type) {
    case "SET_FULL_TIME":
      return {
        ...state,
        fullTime: action.payload,
      }
    default:
      throw new Error(
        `Action type: ${action.type} was not accounted for in myReducer`
      )
  }
}
```

This is where the `useReducer` really starts to shine - we can define exactly
how we want our state to update for any action we define. What's also great is
that we can define actions for each state update so that we can take our complex
state and manage it with less headache.

To make sure we don't introduce typos into our code, it's common to define the
action types as constants and make **action creators** for each one. An action
creator is a function which returns one of our action object. For our
`SET_FULL_TIME` action, we could do the following.

> Note: if the
> [arrow function](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Functions/Arrow_functions)
> syntax seems odd, these arrow functions implicitly return the value following
> the arrow.

```javascript
// here's an action type, defined as a constant to prevent typos
const SET_FULL_TIME = "SET_FULL_TIME"

// here's our action creator, which returns an action object for our `fullTime` value
const setFullTime = fullTime => ({
  type: SET_FULL_TIME,
  payload: fullTime,
})

// we could then use that action creator like this
dispatch(setFullTime(true))
```

## tl;dr

So now we can see the full picture. The useReducer takes a **reducer function**
and an **initial state**, it then returns a **state** and a function to
**dispatch** actions. Any action objects that get dispatched will run through
the reducer function to return a new state.

Once we grasp this behaviour, and how the flux pattern works with React, we can
start to make sense of how to locate state in our React apps, and manage that
state effectively. useReducer is an incredibly useful React API, and I hope this
article encourages the React community to make more use of it!
